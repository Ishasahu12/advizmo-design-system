Production-ready design system for millions of users. Every pixel earns its place.

Section 01

Core Principles

Five non-negotiable principles that guide every design decision in Advizmo.

👁

Clarity Over Cleverness

Every element must be immediately understandable. No ambiguous icons. No uncertain labels. When a user sees a number, they must know exactly what it means. Financial data demands absolute clarity — there is no room for interpretation. If a design requires explanation, it is wrong. A first-time user encountering any screen should know what to do within two seconds.

🛡

Trust Through Transparency

Every recommendation must explain Why (the reasoning behind it), How (the mechanism of action), What changes (the specific modifications), and What happens next (the expected outcome). Never hide fees. Never obscure terms. Never bury important information behind taps or scrolling. Users trust Advizmo because we show them everything — the good and the bad.

♻

Scale Through Reusability

Every component must work across all contexts within the app. Design for the worst case first: support text in German, numbers as large as $1,000,000,000, names with 40+ characters. If a component only works with short English labels, it is not production-ready. Reusability eliminates inconsistency and reduces engineering cost.

🧠

Reduce Cognitive Load

Show only what is needed at each moment. Use progressive disclosure to reveal complexity gradually. Maintain consistent patterns across similar interactions. Financial apps are inherently complex — our job is to absorb that complexity so users can make confident decisions without mental fatigue.

🚀

Ship Tomorrow

Every design must be implementable by engineering with existing tools and tokens. If a design requires a custom animation framework or a new library, reconsider. Follow Apple Human Interface Guidelines and meet WCAG 2.2 AA accessibility standards. The best design is the one that ships to users.

Section 03

Component Rules

Strict guidelines for every UI component. Follow these rules without exception.

Cards

Maximum 3 hierarchy levels within a single card. Use consistent 16px internal padding. Apply 16px border radius. Cards must not contain more than one primary action. Secondary actions use text links. Never nest cards inside cards.

Buttons

Primary buttons use --blue (#0a84ff) with white text. Secondary buttons are outlined with no fill. Destructive buttons use --red (#b91c1c). Disabled buttons reduce opacity to 0.5. Loading buttons show a shimmer animation. All buttons must be at least 44px tall for touch targets.

Inputs

Fixed height of 48px for all text inputs. Focus state adds a 3px blue ring offset. Error state switches border to red and shows helper text below. Never use placeholder as label — always use floating labels or visible labels above.

Section 04

Interaction Rules

Consistent motion and feedback patterns. Every interaction must feel intentional and predictable.

Expand / Collapse

Use 200ms ease-out timing. Animate both height and opacity simultaneously. Content fades in as it expands. On collapse, content fades out before height reduces. Never animate only height without opacity — it creates a jarring snap. Always use overflow:hidden during animation to prevent content from spilling out.

⬇
200ms ease-out • height + opacity

Loading States

Skeleton screens use a pulse animation at 1.5s infinite cycle. Show 3–5 skeleton lines matching expected content layout. Never show a spinner for content that will appear in under 500ms. For operations taking 500ms–2s, show a subtle progress indicator. For operations over 2s, show a progress bar with percentage.

Pull-to-Refresh

Minimum animation duration of 300ms to feel intentional. Show a circular progress indicator. Content shifts down 60px during pull. On release, snap back with spring animation. Never refresh silently — always show visual feedback that data is being updated.

🔄
300ms minimum • spring snap-back

Dismiss

Swipeable items slide right and fade out over 200ms. Show a destructive action (delete) or restore action (undo) on swipe. Never dismiss permanently without confirmation for destructive actions. Provide an undo window of 5 seconds for accidental dismissals.

👉
slide right + fade • 200ms

Success Feedback

Show a green checkmark icon that scales up from 0.8 to 1.0 over 300ms. Follow with a brief toast notification. For financial actions (transfers, investments), show a confirmation screen with transaction details. Never rely solely on color to indicate success — always pair with an icon and text.

✅
green checkmark • 300ms scale-up

Failure Feedback

Show a red X icon with a horizontal shake animation (translateX oscillation over 200ms). Display a clear error message explaining what went wrong and how to fix it. Provide a retry action button. Never show generic "Something went wrong" — always explain the specific failure.

Section 05

Accessibility Rules

WCAG 2.2 AA compliance is mandatory. Every screen must work for every user.

  • 1
    Color Contrast: Normal text must meet 4.5:1 contrast ratio against background. Large text (18px+ bold or 24px+ regular) must meet 3:1 ratio. Never use color alone to convey information — always pair with icons, text, or patterns.
  • 2
    Touch Targets: All interactive elements must be at least 44×44 points. Buttons, links, and tappable areas must not overlap. Maintain minimum 8px spacing between touch targets. Icons used as buttons must have adequate hit area padding.
  • 3
    Dynamic Type: All text must scale with iOS Dynamic Type settings. Layouts must reflow gracefully at 200% text size. No text should be truncated at default size — truncation is only acceptable for overflow at larger sizes. Test at x1, x2, and x3 text sizes.
  • 4
    VoiceOver: Every image must have meaningful alt text or be marked decorative. All interactive elements must have descriptive labels. Complex components (charts, graphs) must provide summary descriptions. Navigation order must follow logical reading order. Hints must explain non-obvious interactions.
  • 5
    Reduced Motion: Respect the "Reduce Motion" system setting. Replace animations with opacity fades or instant state changes. Never auto-play animations. Carousel animations must pause. Parallax effects must be disabled. Loading spinners may remain but should be subtle.
  • 6
    Localization: Support RTL layouts for Arabic and Hebrew. Format numbers according to locale (1,000.00 vs 1.000,00). Display currency with proper symbol placement. Format dates based on locale conventions (MM/DD/YYYY vs DD/MM/YYYY). All text must be translatable with room for 30% text expansion.
Section 06

Edge Case Rules

Production apps must handle every scenario gracefully. Design for the unexpected.

Data Edge Cases

Zero Balances

Show "$0.00" not "$0" or "—". Include a CTA to add funds. Never hide zero values.

Negative Balances

Display in red with a minus sign. Never use parentheses for negatives in consumer apps. Explain why the balance is negative.

Large Numbers

Format $1,000,000,000 as "$1B". Use compact notation for display, full precision for details. Never let numbers overflow their containers.

Small Numbers

Display $0.01 as "$0.01" not "1¢". For amounts under $1, show full decimal. For percentages, show 1 decimal: 0.1%.

State Edge Cases

Offline

Show cached data with a banner: "You're offline. Data may be outdated." Disable actions that require network. Queue actions for when connection restores.

Loading

Show skeleton screens, not spinners. Match skeleton layout to expected content. Never block the entire screen during loading.

Error State

Show specific error message with retry button. Never show a blank screen. Preserve user input across retries. Log errors for debugging.

Empty State

Always show a helpful empty state with illustration, explanation, and CTA. Never show a blank white screen. Guide users toward their first action.

User Edge Cases

First-Time User

Onboarding must be skippable. Progressive feature introduction. Highlight key actions with subtle coaching marks. Never assume prior knowledge.

Returning User

Resume from last state. Show recent activity. Welcome back messaging. Preserve preferences across sessions.

Expert User

Keyboard shortcuts. Batch operations. Advanced settings accessible but not prominent. Power-user features in context menus.

Frustrated User

Clear error recovery paths. One-tap support access. Undo for destructive actions. Never trap users in a broken state.

Content Edge Cases

Long Text

Truncate with ellipsis at container boundary. Show full text on tap/expand. Never overflow or break layout. Test with German compound words.

Long Numbers

$1,234,567.89 must not overflow. Use responsive font sizing or compact notation. Right-align financial numbers for scanability.

Long Names

"Alexandrovich Alexandrovsky" must fit. Truncate after 25 characters in list items. Show full name on detail view or on hover.

Many Items

Virtual scroll for lists over 50 items. Lazy load images. Progressive loading for infinite scroll. Show count in header.

Technical Edge Cases

Slow Connection

Show connection quality indicator. Degrade gracefully — show cached data first. Reduce image quality on 3G. Preload critical resources.

API Failure

Retry with exponential backoff (1s, 2s, 4s). Show user-friendly error. Never show raw API errors. Preserve user's input.

Partial Sync

Indicate which data is stale. Show "last updated" timestamps. Allow manual refresh. Never mix stale and fresh data without labeling.

Conflicts

Show both versions with clear diff. Let user choose which to keep. Never silently overwrite. Preserve conflict history for 30 days.

Section 07

Financial Trust Rules

Money is personal. Every financial interaction must earn and maintain user trust.

  • 1
    Every number must have a source. Show data provider, update timestamp, and calculation methodology. Never present a number without context. "Balance: $24,583.42" must include "as of [date]" and "from [source]".
  • 2
    Every projection must have a disclaimer. Past performance never implies future results. Show "Projected" label clearly. Include time horizon and assumptions. Show range (best case / worst case) not just a single number.
  • 3
    Currency formatting: Always show 2 decimal places ($1,234.56). Use thousand separators. Right-align in tables. Never show "$1234.56" without commas. For international users, respect locale conventions.
  • 4
    Percentage formatting: Always show 1 decimal place (12.3%). Use "↑" for positive and "↓" for negative. Include time period ("+2.4% this month"). Never show "12%" when it means "12.3%".
  • 5
    Recommendation structure: Every recommendation must clearly explain: Why (the rationale backed by data), How (the specific mechanism), Outcome (what the user gets), Benefit (why it matters), Time (expected timeframe), and Risk (what could go wrong).
  • 6
    Fee transparency: Never hide fees. Show total cost before confirmation. Break down fees by type. Compare to alternatives. Show fee as both dollar amount and percentage. "Free" means free — no hidden charges.
  • 7
    Transaction confirmation: Show full details before final confirmation: amount, fees, recipient, estimated time, and reversibility. Require explicit consent for irreversible actions. Provide a 5-second undo window.
Section 08

Scalability Rules

Design for growth. Every component must work across the full range of conditions.

  • 1
    Text Scaling: Every layout must work at 1x, 2x, and 3x Dynamic Type. Test with the largest accessibility text size. No text should be clipped. Buttons must expand to accommodate larger text. Minimum touch target size must be maintained at all scales.
  • 2
    Data Volume: Every list and grid must work with 1 item and 100 items. Single item must not look broken or sparse. 100 items must not cause performance degradation. Use virtual scrolling for lists over 50 items. Lazy load content below the fold.
  • 3
    Device Range: Design for iPhone SE (375px) through iPhone 15 Pro Max (430px). Test on both. Never assume a specific screen size. Use flexible layouts that adapt. Content priority must be maintained on smaller screens — remove chrome, not content.
  • 4
    Number Range: Every financial display must handle $0 and $1,000,000,000. Compact notation for large numbers ($1B). Full precision for small numbers ($0.01). No layout breakage at any value. Right-align numbers for scanability.
  • 5
    Language Range: Every component must accommodate text expansion (German: +30%, Finnish: +40%). Truncation must be graceful. RTL support must be built in from the start, not bolted on later.
Section 09

Engineering Implementation

Design and engineering are partners. Every component must be buildable, testable, and maintainable.

Component API

  • →
    Props: Every component accepts: size (sm/md/lg), state (default/focus/error/disabled), variant (primary/secondary/destructive), and data (content payload).
  • →
    Events: Components emit: onPress (primary tap), onLongPress (secondary action), onDismiss (swipe/remove). All events include event metadata for analytics.
  • →
    States: Every component supports three presentation states: loading (skeleton), error (retry), and empty (illustration + CTA). These are built into the component, not bolted on.

State Management

  • →
    Server State: Cache API responses. Show stale data while refreshing. Optimistic updates for mutations. Rollback on failure. Never show loading for cached data.
  • →
    Client State: Preserve form inputs across navigation. Persist user preferences locally. Sync state across tabs where relevant. Handle state restoration after app backgrounding.

Performance

  • →
    Skeleton Loading: Show skeleton screens within 100ms of navigation. Match skeleton dimensions to expected content. Never show spinner where skeleton is appropriate.
  • →
    Lazy Loading: Load images below the fold on scroll. Defer non-critical JavaScript. Prefetch likely next screens. Use IntersectionObserver for scroll-triggered loads.
  • →
    Memoization: Memoize expensive computations. Use React.memo for pure components. Avoid unnecessary re-renders. Profile with React DevTools before optimizing.
  • →
    Virtual Scrolling: Use virtualized lists for anything over 50 items. Maintain scroll position during updates. Handle dynamic height items. Preserve focus during scrolling.

Testing

  • →
    Unit Tests: Every component must have unit tests for all props, states, and events. Test loading, error, and empty states. Test accessibility attributes. Aim for 80%+ coverage.
  • →
    Integration Tests: Test component composition. Test user flows end-to-end. Test with mock API responses including errors and delays. Test navigation and state persistence.
  • →
    Visual Regression: Snapshot test every component in all variants. Test at 1x, 2x, and 3x text sizes. Test on iPhone SE and iPhone 15 Pro Max. Test in RTL layout. Automate in CI.
  • →
    Accessibility Tests: Automated Axe testing on every screen. Manual VoiceOver testing on every flow. Test with Dynamic Type at maximum size. Test with Reduce Motion enabled. File accessibility bugs as P0.
